# DynamoDB design research

This note explores DynamoDB table/key design for the BTC price guessing game. The current spec has already chosen one durable player record per `x-user-id`, with score and active guess in the same item.

## Goals

- Persist score and active guess across browser sessions.
- Allow exactly one active guess per anonymous user.
- Make guess creation and resolution atomic with conditional updates.
- Avoid DynamoDB transactions for the MVP.
- Avoid durable price snapshot records because snapshot validation uses signed snapshot tokens.

## Recommended design

Use one DynamoDB table for player state.

Table name:

- `BtcGuessPlayers`

Primary key:

- Partition key: `userId` string.
- No sort key.

Item shape:

```json
{
  "userId": "anonymous-browser-uuid",
  "score": 0,
  "activeGuess": {
    "id": "guess-uuid",
    "direction": "UP",
    "startPriceUsd": 67482.15,
    "createdAt": "2026-06-25T12:00:10.000Z",
    "eligibleAt": "2026-06-25T12:01:10.000Z",
    "priceSnapshotObservedAt": "2026-06-25T12:00:00.000Z"
  },
  "createdAt": "2026-06-25T12:00:00.000Z",
  "updatedAt": "2026-06-25T12:00:10.000Z"
}
```

When there is no active guess, omit `activeGuess` instead of setting it to `null`. This makes DynamoDB conditional expressions simpler.

## Attribute notes

- `userId`: copied from `x-user-id`; this is the partition key.
- `score`: integer score, starts at `0`.
- `activeGuess.id`: unique id generated by the backend when the guess is accepted.
- `activeGuess.direction`: `UP` or `DOWN`.
- `activeGuess.startPriceUsd`: decoded from the signed price snapshot token.
- `activeGuess.createdAt`: backend time when the guess was accepted.
- `activeGuess.eligibleAt`: `createdAt + 60 seconds`, stored for client display and backend checks.
- `activeGuess.priceSnapshotObservedAt`: optional but useful for debugging which displayed snapshot the user bet against.
- `createdAt`: first time the player item was created.
- `updatedAt`: last backend mutation time.

## Conditional write patterns

### Lazy player creation

On first `GET /state`, create the player item only if it does not exist.

Condition:

```text
attribute_not_exists(userId)
```

If the condition fails, read and return the existing item.

### Create guess

Use one conditional update against the player item.

Condition:

```text
attribute_not_exists(activeGuess)
```

Update:

```text
SET activeGuess = :guess,
    updatedAt = :now
```

If the player item may not exist yet, either create it before the update or use an update that initializes `score`, `createdAt`, and `activeGuess` together when the item is absent. The simpler MVP path is to ensure player creation during `GET /state`, then still defensively handle missing items in `POST /guesses`.

Failure mapping:

- Conditional failure because `activeGuess` exists: `409 ACTIVE_GUESS_EXISTS`.

### Resolve guess

Use one conditional update against the player item.

Condition:

```text
activeGuess.id = :guessId
```

Update on resolution:

```text
SET score = score + :scoreDelta,
    updatedAt = :now
REMOVE activeGuess
```

Failure mapping:

- No active guess or guess id mismatch: `409 NO_ACTIVE_GUESS` or a safe idempotent state refresh.

This prevents repeated or concurrent resolution requests from scoring the same guess twice.

## Option comparison

### Option A: One player table, no sort key

Pros:

- Simplest schema.
- Single item read/write per user state operation.
- Conditional updates are straightforward.
- No transactions needed.
- Matches the current MVP scope.

Cons:

- No history of past guesses.
- No leaderboard/query patterns.
- Harder to audit resolved games.

Mitigation:

- History and leaderboard are explicit non-goals.
- Add a separate guess-history table later only if requirements change.
- Keep enough active guess metadata to debug current pending guesses.

Scaling note:

- This option scales well for the current access pattern: read or update one player by `userId`.
- It does not scale into cross-user features by itself. Leaderboards, analytics, admin review, and guess history would require adding new tables or indexes later.
- Adding a separate guess-history table later is manageable because the current player item can remain the source of truth for score and active guess. New resolved guesses can be written to the history table after the feature is introduced.
- Backfilling old guess history would not be possible because this MVP design does not store resolved guesses. If future audit history matters, choose a history design before launch.
- Adding a leaderboard later is a moderate change: score is already stored, but querying top scores requires either a new leaderboard table, a GSI designed for score ranking, or an asynchronous projection.
- The migration risk is low if future features only need data going forward. The risk is high only if future requirements need historical guesses that were never stored.

Recommendation:

- Use this option.

### Option B: Single-table design with `PK` and `SK`

Example:

- `PK = USER#<userId>`
- `SK = PROFILE` for player state
- `SK = GUESS#<guessId>` for guess history

Pros:

- Easier to add history later.
- Supports richer query patterns.
- Common pattern for larger DynamoDB apps.

Cons:

- More complex than needed.
- Active guess uniqueness may require extra conditional item or transaction design.
- More cognitive load for an MVP exercise.

Mitigation:

- If chosen, still keep active guess on the profile item for atomic score/active state updates.
- Write guess-history items only after resolution, not during MVP guess creation.

Recommendation:

- Do not use for the initial implementation.

### Option C: Separate player and guess tables

Pros:

- Clean separation between player state and guesses.
- Natural place for guess history.
- Easier audit trail.

Cons:

- Requires transactions or careful multi-write consistency to enforce one active guess and score updates.
- More moving parts.
- Not needed for non-goal features.

Mitigation:

- Use DynamoDB transactions if requirements expand to durable guess history.
- Keep the player item as the source of truth for active guess even if guess history is added later.

Recommendation:

- Avoid for the MVP.

## Indexes

No secondary indexes are needed for the MVP.

Reasons:

- The app only reads by `x-user-id`.
- Leaderboards and cross-user analytics are non-goals.
- Guess history is not required.

## TTL

No TTL is required for player items because score and active guess must persist across sessions.

No snapshot TTL is required because signed snapshot tokens avoid durable snapshot records.

If a future guess-history table is added, TTL can be considered for old history only if long-term auditability is not needed.

## Open implementation details

These can be finalized during backend implementation:

- Exact physical table name per environment, for example `epilot-btc-guess-players-dev`.
- Whether `createdAt` is set during first `GET /state` or first mutating request.
- Whether failed resolution due to a missing active guess returns `409` or returns refreshed `GET /state` data with neutral feedback.
- Whether to store `lastResolvedGuess` briefly for clearer idempotent resolve UX.

## Recommendation summary

Use a single DynamoDB table with `userId` as the only key. Store `score` and `activeGuess` in the same item. Omit `activeGuess` when no guess is pending. Use conditional updates for guess creation and resolution. Do not add secondary indexes, snapshot records, history tables, or transactions for the MVP.
